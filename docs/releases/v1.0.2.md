# Boilerplate v1.0.2

Release axis: boilerplate

Catalog contracts: core-v1 and index-v2 (unchanged)

Manifest and provider contracts: manifest-v1 and provider-v1 (unchanged)

Profiles: bootstrap, standard (default and legacy), and scaled (unchanged)

Compatible soku: soku/v0.1.2 remains readable for legacy lifecycle state;
soku/v0.1.3 is the recommended full-lifecycle client

Migration from previous release: v1.0.1 to v1.0.2 changes the canonical
downstream CI source and the scaled AGENTS.md source; no removals, ownership
transitions, or mergeable-path changes

Provider compatibility: provider-v1 unchanged; optional deprecated legacy ref
behavior is unchanged

Companion tag: soku/v0.1.3

This corrective PATCH release makes the canonical downstream workflow source
authoritative for CLI rendering. Explicit job markers select only the requested
JavaScript, Python, Go, Java/Spring, or configuration job, and malformed or
ambiguous source markers fail closed before any project mutation. The source
uses formatter-compatible version scalars and remains readable as a manually
copyable starter workflow.

The scaled profile also carries the repository's tool-invocation and batched
verification guidance in `AGENTS.md`. This is an operational documentation
change only; it does not alter catalog schemas, profile composition, file
ownership, mergeable paths, or delivery behavior.

The migration from `v1.0.1` changes exactly these catalog-managed sources:

- `templates/_shared/ci/downstream-ci.yml`
- `AGENTS.md` in the scaled profile layer

There are no removals, ownership transitions, mergeable-path changes, or manual
actions for a clean `v1.0.1` snapshot. Existing downstream modifications to
core-managed files continue to use the lifecycle's drift and conflict rules.
The CLI compatibility path is explicit: `soku/v0.1.2` can inspect existing
manifest-v1 state and legacy releases, while `soku/v0.1.3` is required for the
source-authoritative generated workflow and full fresh `init --verify` smoke.

Existing `v1.0.0` and `v1.0.1` tags remain immutable historical baselines. The
corrective pair does not move, delete, or reuse either release.

Release validation requires Markdown/YAML/action semantics, release-note and
tag regression, sync parity, the full Go unit/race/vet/format/import/lint and
lifecycle/package gates, all runtime-template and hosted gates, the required
security workflow, and the aggregate `Validation Gate`. Public smoke must
cover fresh four-application-stack `init --yes --verify`, status, same-release
diff, no-op upgrade, and migration from both `v1.0.0` and `v1.0.1`.

Release preflight uses `boilerplate-tag` set to `v1.0.2` and `cli-tag` set to
`soku/v0.1.3`; its publish job must be skipped. Delivery requires a separately
approved signed annotated `v1.0.2` tag on the reviewed source commit. The
boilerplate GitHub Release receives no CLI archives or checksum asset.
