# Boilerplate v1.0.3

Release axis: boilerplate

Catalog contracts: core-v1 and index-v2 (unchanged)

Manifest and provider contracts: manifest-v1 and provider-v1 (unchanged)

Profiles: bootstrap, standard (default and legacy), and scaled (unchanged)

Compatible soku: soku/v0.1.3

Migration from previous release: v1.0.2 to v1.0.3 changes only the
JavaScript/TypeScript/Node `.prettierignore`; no removals, ownership
transitions, or mergeable-path changes

Provider compatibility: provider-v1 unchanged; optional deprecated legacy ref
behavior is unchanged

Companion tag: none

This corrective PATCH release fixes fresh multi-application-stack
`init --yes --verify` with the published `soku/v0.1.3` CLI. The JavaScript
formatter previously traversed the generated multi-stack root and rejected the
Go `.golangci.yml` and rendered `.github/workflows/ci.yml`, even though those
files are validated by their own YAML and action semantics gates. The
JavaScript template now excludes those two cross-stack YAML surfaces from
Prettier while preserving their dedicated validation.

The failure was found by the required public smoke after `v1.0.2` and
`soku/v0.1.3` were published. Both tags and Releases remain immutable. The CLI
performed the correct staged verification and atomic rollback, so no CLI code,
manifest schema, catalog structure, provider contract, profile composition,
file ownership, or delivery behavior changes in this release.

The migration from `v1.0.2` changes exactly one catalog-managed source:

- `templates/javascript-typescript-node/.prettierignore`

There are no removals, ownership transitions, mergeable-path changes, or
manual actions for a clean `v1.0.2` snapshot. Existing downstream modifications
continue to use the lifecycle's drift and conflict rules. `soku/v0.1.3` can
inspect and migrate existing manifest-v1 state from `v1.0.0`, `v1.0.1`, and
`v1.0.2` to this release.

Existing `v1.0.0`, `v1.0.1`, and `v1.0.2` tags remain immutable historical
baselines. This release does not move, delete, or reuse them and does not create
a new CLI release.

Release validation requires Markdown/YAML/action semantics, release-note and
tag regression, sync parity, the full Go unit/race/vet/format/import/lint and
lifecycle/package gates, all runtime-template and hosted gates, the required
security workflow, and the aggregate `Validation Gate`. Public smoke must cover
fresh four-application-stack `init --yes --verify`, status, same-release diff,
no-op upgrade, and migration from `v1.0.0`, `v1.0.1`, and `v1.0.2`.

Release preflight uses `boilerplate-tag` set to `v1.0.3` and `cli-tag` set to
`soku/v0.1.3`; its publish job must be skipped. Delivery requires a separately
approved signed annotated `v1.0.3` tag on the reviewed source commit. The
boilerplate GitHub Release receives no CLI archives or checksum asset.
