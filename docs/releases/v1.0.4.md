# Boilerplate v1.0.4

Release axis: boilerplate

Catalog contracts: core-v1 and index-v2 (unchanged)

Manifest and provider contracts: manifest-v1 and provider-v1 (unchanged)

Profiles: bootstrap, standard (default and legacy), and scaled (unchanged)

Compatible soku: soku/v0.1.4

Migration from previous release: v1.0.3 to v1.0.4 changes only the Python
`pyproject.toml`; no removals, ownership transitions, or mergeable-path changes

Provider compatibility: provider-v1 unchanged; optional deprecated legacy ref
behavior is unchanged

Companion tag: none

This corrective PATCH release isolates Python lint and type checking from a
co-generated JavaScript dependency tree. Required public smoke for immutable
`v1.0.3` found that `ruff check .` traversed
`node_modules/flatted/python/flatted.py` during four-application-stack
`init --yes --verify`. The CLI correctly aborted before applying the manifest.

The Python template now excludes `node_modules` from Ruff and mypy discovery.
Python `src` and `tests` remain fully checked, while JavaScript dependencies
remain governed by their own lint, typecheck, test, build, audit, and formatting
commands. Catalog, manifest, profile, provider, ownership, recovery, and
delivery contracts do not change.

The migration from `v1.0.3` changes exactly one catalog-managed source:

- `templates/python/pyproject.toml`

There are no removals, ownership transitions, mergeable-path changes, or
manual actions for a clean `v1.0.3` snapshot. Existing public tags and Releases
remain immutable. Published `soku/v0.1.4` can initialize and migrate to this
single-axis boilerplate correction without a new CLI release.

Verification requires the rendered multi-stack boundary regression, Python
Ruff/mypy/Black/pytest checks with a populated JavaScript `node_modules`, full
repository and runtime-template validation, three-OS lifecycle, five-target
package snapshot, dependency and vulnerability audits, sync parity, and the
aggregate `Validation Gate`. Public smoke must cover fresh four-application-
stack `init --yes --verify`, status, same-release diff, no-op upgrade, and
migration from `v1.0.3` to `v1.0.4`.

Release preflight uses `boilerplate-tag` set to `v1.0.4` and `cli-tag` set to
`soku/v0.1.4`; its publish job must be skipped. Delivery requires separately
approved signed annotated `v1.0.4` on the reviewed source commit. The
boilerplate GitHub Release receives no CLI archives or checksum asset.
